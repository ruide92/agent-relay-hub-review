## P0 — 阻塞项（Phase 1 前必须关闭）

### C1 ｜ 许可证维度 ｜ 定义 ARH 自身许可证与插件许可兼容矩阵
- **问题（G28/G29）**：全文规定依赖与插件的许可，却从未说明 ARH 自己开源还是专有；无插件许可兼容性矩阵（GPL/AGPL 与专有核心冲突未处理）。没有 ARH 自身许可，插件生态与商业化两章悬空。
- **修改建议**：
  1. 新增 §23.4「ARH 许可模型」：明确 ARH 核心采用何种许可（推荐 Open Core：核心引擎 Apache-2.0 / 商业版专有），并说明开源与商业共存方式。
  2. 新增「插件许可兼容矩阵」：规定当 ARH 核心为 X 许可时，可加载插件的许可白名单/黑名单；明确 **GPL/AGPL 插件不得加载进专有核心**，仅 MIT/Apache/BSD/自有许可插件允许。
  3. 在 §24.1 Source of Truth 增加 `LICENSE.md` 与 `THIRD_PARTY_LICENSES.md` 为必含文件。
- **验收标准**：V1.1 存在 ARH 自身许可声明 + 插件许可兼容矩阵；任一插件 PR 的许可若落入黑名单，合约测试/CI 直接失败。

### C2 ｜ 确认点 8 / 可实施性 ｜ 把 MVP 真正收窄
- **问题（G12）**：§19.1「必须实现」含内核+模拟+4 个真实适配器+多审核委员会+全套插件协商+E2E，已是一个准平台，违反 §26 原则 1「首个实现必须窄」。
- **修改建议**：
  1. 重写 §19.1 为 **MVP-core（真·窄）**：
     - 内核状态机 + 事件账本（SQLite）+ 政策引擎；
     - **1 个模拟适配器**（跑通整条闭环与全部硬限制）；
     - **1 个真实适配器：Git**（仅隔离测试仓库）；
     - 终态状态机 + 防重/重试/心跳/预算/硬循环限制；
     - 托盘状态/暂停/急停 + 零打扰摘要（可先用日志/RPC 摘要，不必先做 GUI 通知适配器）；
     - 完整 E2E 自动化测试。
  2. 浏览器 / Windows UIA / 通知 / 双 reviewer+仲裁+验证器 → 从 §19.1 移出，明确归入 Phase 2 / Phase 4。
  3. 改写 §18.2：将「首批模板」改为「**未来模板候选**」，并加注「MVP 仅实现软件开发闭环（代码类），其余模板在对应 Phase 验证证据类型后接入」。
- **验收标准**：§19.1 的「必须实现」条目数显著收敛；Phase 1 交付不含浏览器/UIA/通知适配器；§18.2 不再把非代码模板称为首批。

### C3 ｜ 确认点 2/7 / 安全 ｜ 策略完整性提升为硬性安全需求
- **问题（G7/G16）**：§22 风险表提到「政策签名、只读加载」，但 §15 安全模型未将其列为强制条款；策略可被 AI 或本地篡改而无加载校验。
- **修改建议**：
  1. 在 §15 新增「15.5 策略完整性」：政策文件须带哈希（建议同时支持签名）；ARH 启动时校验哈希/签名；不匹配则拒绝启动并进入 **safe-mode（只读、不自动执行任何 A2+ 动作）**，仅允许人工介入修复政策。
  2. 在 §24.2 变更流程补：「政策文件变更必须独立 commit 且触发哈希重算；加载校验不通过禁止运行。」
- **验收标准**：篡改政策文件后 ARH 启动即拒/进 safe-mode；自动化测试含「政策被改→启动失败」用例。

### C4 ｜ 确认点 6 / 自动闭环 ｜ 防重/循环计数器必须持久化
- **问题（G 计数器）**：§9 说「不隐式重置计数」，但未规定计数器存于事件账本/SQLite，崩溃重启若用内存将归零，使防重与硬循环限制失效。
- **修改建议**：
  1. 在 §9 额外规则补：「所有去重键（SHA、message_id）、尝试次数、修复轮数、仲裁重试、重新设计轮数均为**持久状态**，写入事件账本/SQLite；崩溃恢复后从账本重建，**不得归零**。」
  2. 在 §12.1/§12.2 明确这些计数器属于事件溯源的可重放状态。
- **验收标准**：注入「处理中途杀死进程」故障，重启后同一 SHA/消息不再重复处理，循环计数延续而非从 0 开始（§21.2 #2/#4 的延伸测试）。

### C5 ｜ 确认点 3/7 / 全权代理 ｜ 授权天花板进能力协商 + 政策引擎唯一裁决
- **问题（G6）**：§10.2 能力协商无 `authorization_tier_ceiling`；越界路由依赖协调器自觉，缺「政策引擎唯一裁决、越界即拒」的硬机制。
- **修改建议**：
  1. §10.2 能力清单增加：`authorization_tier_ceiling`（该适配器被预授权的最高档位 A0–A5）。
  2. 新增规则：「**政策引擎是唯一授权裁决者**。任何超出角色预授权档位或适配器 `authorization_tier_ceiling` 的动作请求，政策引擎直接拒绝并写审计；协调器与适配器**无权申诉或重试该动作**。」
  3. §15.1 一次性能力令牌须显式携带其档位上限。
- **验收标准**：向仅清档 A2 的适配器发送 A4 动作 → 被政策引擎拒绝并留痕；合约测试覆盖越界拒绝。

### C6 ｜ 确认点 1 / 产品完整性 ｜ 增设「政策缺口」逃逸终态
- **问题（G1）**：系统绝不问用户，但「既非安全、又无现成政策」的新型情况无出口，易过度阻断或盲猜。
- **修改建议**：
  1. §8.2 终态清单新增 `BLOCKED_NEEDS_POLICY_DECISION`：含义为「无匹配政策，非安全风险，交由次日摘要进入人工政策决策」；明确它与 `BLOCKED_REDESIGN_REQUIRED`（技术性重设计）的区别。
  2. §4.3 增加一行：「出现未覆盖的新型非安全决策 → 进入 `BLOCKED_NEEDS_POLICY_DECISION`，不实时弹窗、不猜测。」
  3. 附录 A 补：「系统宁可进入需次日决策的安全终态，也不向用户实时发送选择题。」
- **验收标准**：构造一个政策未覆盖的非安全情景，系统进入 `BLOCKED_NEEDS_POLICY_DECISION` 且仅在次日摘要出现，运行期无弹窗。

### C7 ｜ 确认点 8 / 自动闭环 ｜ 定义非代码证据或显式推迟
- **问题（G9）**：§18.2 列文档/研究/数据模板为首批，但 verifier/evidence 全是代码中心，非代码「确定性证据」未定义。
- **修改建议**（二选一，推荐 A）：
  - **A（推荐）**：§18.2 改标「未来候选」；在 MVP 范围（C2）明确「MVP 仅代码类闭环」，并加注「非代码模板在定义出对应证据类型（如结构化 schema 校验、外部 API 状态查询、人工签名确认）后于后续 Phase 接入」。
  - **B**：为非代码流程定义至少一种非平凡证据类型与对应 verifier 规范，纳入 MVP 或紧邻 Phase。
- **验收标准**：V1.1 不再暗示 MVP 同时覆盖代码与非代码闭环；若选 B，则给出可测试的非代码证据定义。

### C8 ｜ 安全 / 可实施性 ｜ 备份恢复与审计锚定
- **问题（G2/G17）**：MVP 的 SQLite 备份/恢复、WAL 损坏恢复、产物回滚未定义；本地审计哈希链可被整体重写后重算，缺锚点。
- **修改建议**：
  1. §12 新增「12.4 备份与恢复」：MVP 至少定义 SQLite 周期性备份 + 产物仓库备份 + 恢复演练步骤；明确「恢复后必须重放事件账本校验一致性」。
  2. §12.2 补：「本地审计哈希链为尽力而为；Team/Enterprise 档位须接入外部只追加存储或周期签名锚定，防止本地篡改。」
- **验收标准**：文档含可执行备份/恢复步骤；审计锚定在高档位的要求被显式记录。

## P1 — 本版应补强的硬机制 / 默认值

### C9 ｜ 确认点 6 ｜ 补齐默认数值与多层预算
- **问题**：§9「最大执行时间」无默认值；预算仅「按工作流」。
- **修改建议**：
  - 给每 job 墙钟上限默认值（如 24h）、每工作流总上限、并新增**每日 + 全局预算天花板**。
  - 在 §9 表格补充「默认上限」列具体数值，标注可在政策覆盖。
- **验收标准**：所有 §9 上限项均有默认数值；预算分工作流/每日/全局三层。

### C10 ｜ 确认点 2 ｜ 无备用适配器与切换抖动防护
- **问题**：角色无备用时的默认行为未定；两备用都坏可能反复横跳。
- **修改建议**：
  - §4.3/§16.1 补：「某角色**无任何备用适配器**时，主失效直接进对应安全终态，不等待、不询问。」
  - 增「最大切换尝试次数（默认 3）+ 切换退避」，避免备用间抖动。
- **验收标准**：构造「无备用」与「双坏备用」故障，系统分别进安全终态 / 有限次切换后终态，不挂起不抖动。

### C11 ｜ 确认点 4 ｜ 严重度不确定默认保守
- **问题**：reviewer 对同 finding 严重度判定冲突且无可裁决证据时，缺默认。
- **修改建议**：§7.6 补：「严重度无法裁决时，**默认按更高档阻断**（安全优先于通过）。」
- **验收标准**：构造严重度分歧且无证据场景，裁决结果为阻断。

### C12 ｜ 确认点 5 ｜ 发布晋级只认外部证据
- **问题**：意图正确，但措辞可被误读为「文本完成+部分证据」亦可晋级。
- **修改建议**：§7.6/§8 明确：「release_gate **仅**接受 evidence_verifier 的确定性输出或外部系统可验证状态；**禁止**以任何角色的纯文本『完成』作为晋级依据。」
- **验收标准**：worker 返回「完成」但 verifier 未过 → 必阻断（已有 §21.2 #6，V1.1 写死措辞）。

### C13 ｜ 插件化 ｜ 插件崩溃退避与运行时能力降级
- **问题（G21/G22）**：缺 per-plugin 崩溃退避；运行时能力降级对在途 job 的影响未定。
- **修改建议**：
  - §10.4 增「插件崩溃退避：同一插件 N 分钟内崩溃超 M 次则停用并告警，不无限重启」。
  - §10.2/§16 补：「运行中适配器丢失关键能力（如 execute）→ 立即对依赖该能力的在途 job 触发重路由或安全终态。」
- **验收标准**：插件崩溃风暴被退避遏制；模拟运行时能力降级 → 在途 job 正确重路由/终态。

### C14 ｜ 安全 ｜ 验证器沙箱出网与截图脱敏方法
- **问题（G18/G19）**：verifier 沙箱出网未限制；截图脱敏方法未定义。
- **修改建议**：
  - §15 增：「evidence_verifier 沙箱默认阻断出网、限制文件系统白名单，防止恶意测试外泄。」
  - §12.3 定义脱敏方法（如：敏感字段正则替换 + 默认开启 + 关闭需显式策略授权并留审计）。
- **验收标准**：沙箱出网测试被阻断；脱敏方法文档化且可配置。

### C15 ｜ 可实施性 ｜ 插件 SDK 契约落地 + Phase 0 验收判据
- **问题（G15/G13）**：SDK 仅列能力清单，无接口/错误码/生命周期；Phase 0 无可执行验收。
- **修改建议**：
  - 在 §10 或新 §10.5 给出适配器 SDK 契约骨架（生命周期：load→negotiate→serve→shutdown；错误码；JSON-RPC 方法表），或至少是 ARCHITECTURE/PROTOCOL 文档的明确指针。
  - §20 Phase 0 门禁补可执行判据：交叉引用一致性检查、ADR 决议齐备清单、八维度审查结论归档。
- **验收标准**：存在 SDK 契约文档或明确指针；Phase 0 有可勾选的验收清单。

## P2 — 后续补全项

### C16 ｜ 商业/许可证 ｜ SBOM 工具链与 n8n 许可精核
- **建议**：§23.3 指定 SBOM 标准（如 CycloneDX）与生成时机（CI 门禁）；核对 n8n 实际许可（Sustainable Use License）具体条款，替换笼统「fair-code」表述。

### C17 ｜ 产品完整性 ｜ 并行/依赖模型与可观测性
- **建议**：§13 声明「当前仅单 job，并行为后续」；定义结构化指标/SLO/告警阈值（不止 UI 字段）。

### C18 ｜ 商业化 ｜ 护城河与定价挂钩
- **建议**：补技术/数据/网络效应壁垒思考；定价与价值度量（按席位/工作流/适配器/节省人力）挂钩，反推 MVP 付费功能。

### C19 ｜ 商业化 ｜ Marketplace 前置依赖显式化
- **建议**：§23.1 Marketplace 标注「前置 = Phase 7 之后 + 插件商店 + 签名 + 分账基础设施」，避免被误读为近期能力。

### C20 ｜ 全权代理 ｜ 委托链档位继承
- **建议**：§4.2 补 worker→子代理的档位继承规则，禁止隐性提权。

### C21 ｜ 产品完整性 ｜ 国际化策略
- **建议**：明确 MVP 是否做 i18n、摘要语言策略、时区处理，或在非目标中显式后置。
