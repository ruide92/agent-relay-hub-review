## 0. 总体结论（先看这里）

**设计成熟度：高。** V1.0 的核心哲学——"有界全权代理、零打扰、证据优于自述、明确终态、内核级硬限制"——在同类提案中属于少见地想清楚了边界与失败语义的一份。它不是又一个"连接更多 App"的自动化壳子，而是真正在定义一个**治理层**。

**但存在三类必须在本轮（V1.1）解决的结构性问题：**

1. **MVP 范围过宽，与自身"首个实现必须窄"的原则自相矛盾**（第 8 项确认点未真正满足）。§19.1 的"必须实现"清单同时包含内核、模拟器、Git、浏览器、Windows UIA、通知、双 reviewer+仲裁+验证器、完整插件清单与能力协商、E2E——这已经是一个"准平台"，而非一个"窄闭环"。
2. **ARH 自身的许可证未定义**（许可证维度评级为"不足"）。全文只规定了"依赖用 MIT/Apache/BSD、插件各自声明许可证"，却从未说明 ARH 自己开源还是闭源、第三方插件在什么许可下可被加载。没有这一点，插件生态与商业化都无从落地。
3. **多个"必须由内核执行、提示词无权修改"的机制停留在散文描述，未提升为带具体机制的硬性需求**。最典型的是策略完整性（policy integrity）、防重复计数器的持久化、以及"政策引擎为唯一裁决者"的授权天花板。这些在风险表（§22）里提到了，但没进安全模型（§15）和协议（§10/§11）的强制条款。

**八项重点确认点速览：**

| # | 确认点 | 结论 | 关键缺口 |
|---|---|---|---|
| 1 | 运行期不抛常规选择题 | **满足** | 缺少"政策未覆盖的新型非安全决策"的逃逸终态 |
| 2 | 工具失效自动切换或安全终态 | **满足** | 未配置备用时的默认行为、备用间反复横跳的防护未规定 |
| 3 | 不写死 WorkBuddy/Qoder/ChatGPT/GitHub | **满足** | 角色示例含产品名（作为示例可接受），需保证不进编译常量 |
| 4 | 多审核不用简单多数票 | **满足** | 严重度判定冲突且无可 adjudicate 证据时，缺"不确定即保守（阻断）"规则 |
| 5 | AI 完成声明须外部证据验证 | **满足** | 应把"发布晋级只认 evidence_verifier/外部状态，禁止认文本'完成'"写死 |
| 6 | 防重复/防死循环/预算 | **满足** | 计数器必须持久化（崩溃不重置）；缺默认数值与多层预算 |
| 7 | 全权代理不破预授权边界 | **满足** | 缺"授权天花板进能力协商 + 政策引擎唯一裁决 + 越界即拒"的硬机制 |
| 8 | MVP 足够窄 | **部分满足** | §19.1 过宽，§18.2"首批模板"含非代码流程，与窄 MVP 冲突 |

## 1. 产品完整性

**评级：良好（强项维度，仍有缺口）**

### 优点
- 文档结构完整，从执行摘要、背景、愿景、原则、市场、角色、审核、闭环、限制、适配器、协议、数据、架构、节点、安全、可靠性、UX、配置、MVP、路线、测试、风险、商业化、治理、用例到附录，覆盖了一个自治系统应有的全部面。
- "Source of Truth"（§24.1）与"变更流程"（§24.2）意识成熟，明确"文档冲突按优先级处理，不从聊天恢复产品事实"——这对防止 AI 执行者漂移非常关键。
- 明确区分了"完成"与"合并"（§8.3），把闭环延伸到上线后监控与回滚，而不是止步于 PR。

### 缺口 / 风险
- **G1 政策缺口逃逸态缺失**：提案反复强调"运行期绝不问用户"，这是对的。但它没有定义：当一个**既非安全、又未被现有政策覆盖**的新型情况出现时，系统该怎么办。当前所有终态（§8.2）要么偏安全阻塞、要么偏完成，缺少一个 `BLOCKED_NEEDS_POLICY_DECISION`——即"没有现成政策，但也不危险，交由次日摘要进入人工政策决策，而非实时弹窗、也非盲目猜测"的出口。没有它，系统要么过度阻断、要么被迫猜。
- **G2 备份/恢复未定义**：§12.1 提了 SQLite WAL，但 MVP 的备份、WAL 损坏恢复、产物仓库的备份与回滚完全没提。对"夜间无人值守"产品，这是交付可信度的基础。
- **G3 并行/依赖模型缺失**：全文所有流程都是单 job 线性。没有 job 间依赖、fork/join、父-子工作流。MVP 可不需要，但架构（§13）应声明"当前仅单 job，并行为后续"以免被误读为已支持。
- **G4 可观测性/SLO 仅停留在 UI 页面**：§17 列了仪表盘字段，但没有结构化指标、告警阈值、健康 SLO 的定义。对"自主运行"产品，遥测应是产品需求而非仅 UI。
- **G5 国际化未提及**：中文产品，但多语言、时区、摘要语言策略没说。MVP 可后置，但应在非目标或路线中明确。

## 2. 全权代理能力

**评级：良好（档位设计是全文最亮点之一）**

### 优点
- A0–A5 授权档位（§4.2）设计克制且自洽：**A5 默认禁止，且"用户不介入"不导致 A5 自动放行**——这点直接命中"全权代理不是无限权限"。
- §4.3 无人值守默认决策表覆盖了中国、全面：主备切换、心跳、去重、CI 失败退回、冲突仲裁、最大轮数、登录失效、锁屏、预算、不可逆拒绝。这是把"自动驾驶策略"写成机器可执行政策的范例。
- §24.3 把 WorkBuddy/Qoder 等明确定位为"执行者而非所有者"，并列出六条不可为，边界清晰。

### 缺口 / 风险
- **G6 授权天花板未进入能力协商**：§10.2 的能力协商列出了 `send/receive/execute/...` 等技术能力，但**没有 `authorization_tier_ceiling`**——即"该适配器被预先允许执行的最高档位"。后果：协调器可能把一个 A4 动作路由给一个只被清档 A2 的适配器，是否拒绝取决于协调器自觉。应为每个适配器声明其档位天花板，政策引擎据此拒绝越界路由。
- **G7 政策完整性未提升为硬性安全需求**：§22 风险表提到"政策签名、只读加载、代码与配置分权"，但 §15 安全模型未把它作为强制条款。必须规定：加载的策略文件带哈希/签名，加载时校验，不匹配则进入 safe-mode/拒绝启动。否则"AI 修改政策"的防护只是建议。
- **G8 委托链（worker→子代理）的档位继承未定义**：若 worker 再派生子代理，子代理是否继承父档位、能否升档？未说明，存在隐性提权面。

## 3. 自动闭环

**评级：良好（闭环语义强，非代码证据缺位）**

### 优点
- §8.1 主流程 + §8.2 终态清单 + §8.3 上线后闭环，构成了一个真正"有头有尾"的自治环，且明确"没有无限 RUNNING，没有等用户选"。
- 心跳语义（§16.2）正确区分 RUNNING+心跳 / RUNNING+无心跳 / STALLED / 恢复失败，且明确"不因固定 15/60 分钟轮询而重复提交"——直接规避了用户背景里"简单定时轮询可能打断任务"的痛点。
- 崩溃恢复（§16.3）的对账逻辑正确：只有确认未执行才重试，可能已执行的写操作进 `RECONCILIATION_REQUIRED` 而非盲目重发。

### 缺口 / 风险
- **G9 非代码工作流的证据验证未定义（与第 8 项确认点强相关）**：§18.2 把"文档起草—审核—修订""研究收集—交叉核实—报告""数据分析—复核—发布"列为"首批模板"，而 §7/§12 的 verifier 与 evidence 全是代码中心（测试运行器、静态扫描器、沙箱、SHA/CI）。对文档/研究类工作流，"确定性证据"是什么？没有定义。要么在 MVP 明确**只做代码闭环**、把其余模板标为"未来候选"；要么为非代码流程定义至少一种非平凡证据类型（如：结构化 schema 校验、外部 API 状态查询、人工签名确认）。
- **G10 复审范围未规定**：§8.1"回归测试与复审"——修复后的复审是完整重审还是仅针对相关 finding 的部分复审？若是完整重审，可能放大死循环风险；若是部分，需定义"相关"的判定规则。
- **G11 exactly-once 的声明与机制脱节**：§10.2 列了 `supports_exactly_once_claim`，但 ARH 如何"认领"某外部动作确由自己发起、避免把外部系统的动作误判为自己的重试，机制未展开。§16.3 的对账依赖外部系统"可查询实际状态"，应明确哪些外部系统必须提供该能力、否则降级为安全终态。

## 4. 可实施性

**评级：需改进（MVP 过宽是核心问题）**

### 优点
- Phase 0–7 阶段门禁清晰，每个阶段有独立交付与门禁，且明确"WorkBuddy 或其他执行者不得自行跳阶段"。
- 测试层次（§21.1）覆盖单元/属性/集成/合约/E2E/故障注入/安全/回放/升级/长稳，非常完整。
- 技术选型（.NET 8 / SQLite / Playwright / UIA / JSON-RPC）务实，且主动声明"首期不嵌入 n8n/Temporal"，避免 MVP 过重。

### 缺口 / 风险
- **G12 MVP（§19.1）过宽，违反自身"首个实现必须窄"原则（§26 原则 1）**：§19.1 同时要求 Windows 控制中心 + SQLite + 事件审计 + 协议 + 插件清单 + 能力协商 + 模拟适配器 + **Git + 浏览器 + UIA + 通知** 四个真实适配器 + 双 reviewer+仲裁+验证器 + 全套防重/重试/心跳/预算/硬循环 + 托盘 + 零打扰 + E2E。这是 3–4 个 Phase 的工作量被压成一个"MVP"。建议拆成：
  - **MVP-core（真·窄）**：内核状态机 + 事件账本 + 政策引擎 + 1 个模拟适配器 + **1 个真实适配器（Git）** + 终态机 + 防重/循环/预算 + 自动化测试。
  - 浏览器 / UIA / 通知 / 多审核委员会 → 顺延至 Phase 2/4。
- **G13 无工作量/规模估算，Phase 0 自身无验收测试**：§20 Phase 0 门禁仅"设计冲突清零、许可证路线明确"，未定义"如何证明冲突已清零"的可执行判据（如：交叉引用检查、ADR 决议齐备清单）。
- **G14 运行时不跨平台，但演进架构假设跨平台**：首发 Windows/.NET 是刻意选择（§13.3），但 §13.2 NAS 控制面用 PostgreSQL + Web，隐含 Linux 运行。需明确：控制面是否也用 .NET（可跨平台）还是另起栈？若另起栈，则"供应商无关"在运行时层未兑现，应尽早定调。
- **G15 插件 SDK 契约未在本文档落地**：§20 Phase 0 要求"适配器 SDK"，但全文只有能力清单（§10.2），没有 SDK 的接口/错误码/生命周期契约。应至少在 V1.1 给出 SDK 契约的指针或骨架。

## 5. 安全

**评级：良好（强，但几处应提升为硬机制）**

### 优点
- 最小权限（§15.1）按角色发一次性能力令牌，且明确"reviewer 不可写仓库、verifier 不可改源码、adjudicator 不可执行命令"——权限细分到位。
- 凭据（§15.2）不落明文、用 OS 凭据库、只给引用、记录使用不记录值、支持轮换撤销——符合最佳实践。
- Prompt Injection 防护（§15.3）分层、外部文本不能改权限/循环/预算/目标仓库、结果不符 Schema 进隔离——正确。
- 全局熔断（§15.4）含托盘急停、暂停文件、只读状态页、异常风暴自动熔断——覆盖到位。

### 缺口 / 风险
- **G16 策略完整性应成为硬安全需求**（同 G7）：加载校验、safe-mode，必须写进 §15。
- **G17 审计链缺乏锚点**：§12.2 事件溯源用"前序哈希"链，但本地 SQLite 可被整体重写后重算整条链。MVP 可接受"本地不可变"为尽力而为，但必须声明：更高档位（Team/Enterprise）需外部只追加存储或周期签名锚定，否则审计可被本地攻击者篡改。
- **G18 验证器沙箱出网未规定**：evidence_verifier 运行测试/扫描，若被测仓库含恶意测试，可能借验证器外泄。需规定验证器沙箱默认阻断出网、限制文件系统访问白名单。
- **G19 截图脱敏方法未定义**：§12.3"截图默认脱敏并可关闭"，但脱敏是打码、红名还是哈希替换？方法未定，易成合规漏洞。
- **G20 熔断检测阈值缺失**：§15.4"检测异常消息风暴、重复提交或权限绕过后自动熔断"，但"异常/重复"的量化阈值未给，需补默认阈值与可调参数。

## 6. 插件化

**评级：良好**

### 优点
- 适配器类型（§10.1）覆盖 REST/Webhook/浏览器/CDP/UIA/CLI/Git/存储/消息/人工/模拟，面广。
- 能力协商（§10.2）字段设计专业（含 `requires_unlocked_session`、`supports_idempotency`、`max_payload_size` 等）。
- 插件清单（§10.3）强制声明权限、路径/域名白名单、哈希签名、卸载清理——非常完整。
- 插件隔离（§10.4）独立进程、JSON-RPC、禁直连核心库、禁跨插件读凭据、单插件崩溃不拖垮中心、超权由政策引擎拒——隔离模型正确。

### 缺口 / 风险
- **G21 插件崩溃重启风暴未防护**：§10.4 说"单插件崩溃不得拖垮中心"，但没说插件**反复**崩溃时的退避（backoff）。需加 per-plugin crash backoff，避免无限重启风暴。
- **G22 运行时能力降级未处理**：§10.2 把"能力变化"列为健康指标，但未规定：若运行中的适配器**运行时**丢失某项能力（如失去 execute），在途 job 如何反应、是否立即重路由。
- **G23 插件 SDK 契约缺口**（同 G15）：清单要求很多，但"适配器如何被加载/调用/生命周期"的 SDK 契约需在 V1.1 给出。
- **G24 插件版本兼容/回滚未提**：插件升级失败如何回滚、新旧协议共存策略未定义。

## 7. 商业化

**评级：需改进（愿景清晰，路径薄弱）**

### 优点
- 产品层级（Personal/Pro/Team/Enterprise/Marketplace）与可变现能力（§23.2）梳理清晰，差异点"有边界、有证据、有终态"确实是可卖点。
- 许可证原则（§23.3）前瞻性地规避了 fair-code 嵌入商业核心的风险。

### 缺口 / 风险
- **G25 商业化路径是愿景而非计划**：无定价模型、无 GTM、无获客/渠道、无竞争格局下的护城河量化。Marketplace 分成依赖插件商店+签名+分账基础设施，而 §19.2 已把这些**全部推迟**，却未在商业化章节标注"Marketplace  prerequisites = Phase 7 之后且需 X/Y/Z 先行"。应显式声明：商业化能力在 Phase 7 之前仅为方向，不为承诺。
- **G26 护城河薄弱**：核心治理逻辑（终态、证据门禁）若被开源竞品复制，ARH 的差异化会迅速稀释。文档仅有"商标"护城河，缺少技术/数据/网络效应层面的壁垒思考（如：跨工作流的证据库积累、行业模板网络）。
- **G27 定价与价值度量未挂钩**：按席位？按工作流？按适配器？按节省人力？未定，导致 MVP 无法反推"哪些功能决定付费意愿"。

## 8. 许可证

**评级：不足（最关键缺失项之一）**

### 优点
- 核心依赖优先 MIT/Apache/BSD、生成 SBOM、fair-code 不嵌商业核心、插件各自声明许可证、商标不侵权——方向正确。

### 缺口 / 风险
- **G28 ARH 自身的许可证未定义（P0 阻塞）**：全文通篇规定"别人用什么许可、插件声明什么许可"，却**从未说明 ARH 自己开源还是专有**。这是基础性空缺：
  - 若 ARH 专有 → 第三方插件在什么许可下可被加载？GPL 插件能否进专有核心？（不能，需明确禁止）
  - 若 ARH 开源 → 选什么 copyleft/permissive？商业化如何与开源共存（Open Core？）
  - 没有 ARH 自身许可证，**插件生态与商业化都无从设计**。必须在 V1.1 先定 ARH 许可模型。
- **G29 插件许可证兼容性矩阵缺失**：需规定"ARH 核心为 X 许可时，可加载的插件许可白名单/黑名单"（尤其 GPL/AGPL 与专有核心的冲突）。
- **G30 SBOM 工具链未指定**：仅说"生成 SBOM"，未定标准（如 CycloneDX）与生成时机（CI 门禁）。
- **G31 n8n license 表述偏粗**：称其为"fair-code"，实际 n8n 用的是 Sustainable Use License，对"自托管"与"商业嵌入"的边界有具体条款。应核对具体许可文本，而非笼统"fair-code"，以免误判或过度规避。

## 9. 八项重点确认点 — 逐条证据与结论

### 确认点 1：运行期间不得把常规选择题交给用户
- **证据**：§4.1 明确列举 9 类禁止询问项；附录 A-2"系统不向用户发送选择题"；§17.3 零打扰模式；§21.2 验收 #1"用户启动后不需要参与常规选择"。
- **结论：满足。** 散文级约束已非常完整。
- **遗留缺口**：G1（政策未覆盖的新型非安全决策缺逃逸终态）。建议补 `BLOCKED_NEEDS_POLICY_DECISION`，次日摘要进人工，而非实时弹窗或盲猜。

### 确认点 2：工具失效必须自动切换备用或进入安全终态
- **证据**：§4.3"主适配器无响应→按顺序切换备用"；§4.2 A5 受阻→自动"不执行"保留现状；§16.1 基于健康度选主备；§16.2 STALLED→恢复→切换/终态；§22 缓解。
- **结论：满足。**
- **遗留缺口**：
  - 角色**未配置任何备用适配器**时的默认行为未规定 → 必须明确"无备用则直接进对应安全终态，不等待、不询问"。
  - 两个备用都坏时可能反复横跳 → 需"最大切换尝试次数 + 切换退避"，避免抖动。

### 确认点 3：WorkBuddy、Qoder、ChatGPT、GitHub 均不得写死
- **证据**：§6"ARH 不写死产品名称，只定义逻辑角色"；§18.1 纯配置映射；§24.3 明确这些是"执行者而非所有者"，可替换；§25 用例映射仅作示例。
- **结论：满足。**
- **遗留缺口**：需在设计文档/SDK 中加一条硬性约定——"上述产品名只能出现在配置（YAML/示例）与文档示例，**不得**进入协议 Schema 字段或编译期常量"。当前协议（§11）字段为通用（`sender_role`/`repository`），未见写死，符合。

### 确认点 4：多审核角色不得使用简单多数票
- **证据**：§7.1 专节解释"三个相同错误不等于正确、不采用多数通过"；§7.6"意见冲突：仲裁，不做简单多数表决"；§7.2 盲审互不可见。
- **结论：满足。**
- **遗留缺口**：当 reviewer 对**同一 finding 的严重度**判定冲突（一高一低）且无可 adjudicate 的证据时，仲裁员也拿不准 → 缺"不确定即取更保守（阻断）"的默认规则。建议补："严重度无法裁决时默认按更高档阻断。"

### 确认点 5：AI 完成声明必须经过外部证据验证
- **证据**：§7.6"AI 全部通过但 SHA/CI/哈希不一致：阻止"；§8.3"不将代码合并视为完成"；§12.3 证据产物；§21.2 #6"AI 自称成功但测试失败必须阻止"；§24.3"不能在未验证时宣称成功"。
- **结论：满足。**
- **遗留缺口**：建议把发布晋级规则**写死**为"release_gate 只接受 evidence_verifier 的确定性输出或外部系统可验证状态，**禁止**以任何角色的纯文本'完成'作为晋级依据"，消除任何歧义。

### 确认点 6：必须有防重复、防死循环和预算限制
- **证据**：§9 整节——去重表、硬上限、额外规则"达到上限后进入安全终态，不隐式重置计数"；§4.3 预算达限停调用；§21.2 #2/#7。
- **结论：满足（机制设计完整）。**
- **遗留缺口**：
  - **计数器持久化（P0）**：§9 说"不隐式重置"，但崩溃重启后这些计数器必须来自事件账本/SQLite，而非内存。必须明示"防重/尝试/循环计数器属持久状态，崩溃恢复后不归零"。
  - 缺默认数值："最大执行时间"仅写"工作流策略配置"，无默认值；建议给每 job 墙钟上限（如 24h）与每工作流总上限。
  - 预算仅"按工作流"，建议加"每日 + 全局天花板"防跨工作流爆炸。

### 确认点 7：全权代理不得突破预授权安全边界
- **证据**：§4.2 A5 默认禁止且"用户不介入不自动放行"；§4.3"发现不可逆操作：拒绝执行并记录，不询问"；§15 最小权限；§24.3 执行者六条不可为；附录 A-7。
- **结论：满足（意图完全正确）。**
- **遗留缺口（机制提升）**：
  - G7/G16 策略完整性须成硬需求（签名/哈希加载校验）。
  - G6 授权天花板应进能力协商（`authorization_tier_ceiling`），政策引擎为**唯一裁决者**，越界请求直接拒+审计，协调器/适配器无权申诉或重试该动作。

### 确认点 8：MVP 必须足够窄，不能一开始开发万能平台
- **证据（意图）**：§3.3 非目标"首期不实现万能拖拽平台"；§19.2 暂不实现清单；§26 原则 1"首个实现必须窄"；§27"先内核+模拟+隔离仓库闭环，再验证一个桌面/浏览器/Git"。
- **证据（冲突）**：§19.1"必须实现"清单过宽（见 G12）；§18.2 把文档/研究/数据等非代码模板列为"首批模板"。
- **结论：部分满足——哲学正确，但 §19.1/§18.2 与原则自相矛盾，需 V1.1 修订为真正的窄 MVP。**
- **这是八项里最该优先修的一项。**

## 10. 审查小结

V1.0 是一份**思想成熟、边界意识强**的产品书，尤其在"零打扰、有界全权、证据门禁、硬循环限制"上超出普通自动化提案。它真正的薄弱不在"想没想清楚"，而在三处：

1. **说了没钉死**：若干内核级硬机制（策略完整性、计数器持久化、政策引擎唯一裁决）仍停留在散文/风险表，未成为强制条款。
2. **窄了没守住**：MVP 范围与自身"窄实现"原则冲突，是最大的实施风险。
3. **自己的许可没定**：ARH 自身许可证缺失，使插件与商业化两章悬空。

上述均可在 V1.1 通过文档修订解决，无需推翻架构。详细修改清单见 `V1.1_REQUIRED_CHANGES.md`，实施就绪评估见 `IMPLEMENTATION_READINESS_REPORT.md`。
