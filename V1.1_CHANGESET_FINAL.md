## P0 — 阻塞项

### F-C1 ｜ 许可证模型 ｜ 目标 §23 / §24.1
- **最终措辞方向**：ARH 产品源码在 Phase 0–4 保持私有、专有；暂不承诺 Open Core；核心依赖优先 MIT / Apache-2.0 / BSD；GPL / AGPL / SSPL / BUSL / Sustainable Use License 等组件在完成逐项法律兼容审查前不得随产品捆绑分发；Phase 7 产品化前重新评估开源 / Open Core / 全专有商业模式。不得采用"核心 Apache-2.0 + 商业版专有"作为既定结论。
- **验收标准**：V1.1 含明确许可声明（非空白待定）；含"受限许可组件禁止捆绑"条款；Source of Truth 增加 `LICENSE.md` 与 `THIRD_PARTY_LICENSES.md`。

### F-C2 ｜ MVP 收窄与阶段重定义 ｜ 目标 §19 / §20 / §26
- **最终措辞方向**：区分 `Core Prototype`（内核原型）与 `Product MVP`（具备真实产品价值的首个闭环）。Phase 1 仅含状态机、SQLite 事件账本、政策引擎、模拟适配器、模拟工作流、防重/预算/循环限制/崩溃恢复/终态测试；**不含任何真实适配器**。Phase 2 = 真实工具可行性探针（本地 Git、Qoder CLI/SDK、Playwright、UIA、锁屏/UAC/焦点/会话状态），不交付正式产品适配器。Phase 3 = Phase 2 探针通过门禁后，才实现正式适配器和第一个真实代码开发闭环。Phase 4 = 多审核/证据验证/仲裁/夜间无人值守。
- **验收标准**：§19.1 删除所有真实适配器条目；§20 各 Phase 交付与门禁按上述重排；任何"Phase 1 含真实 Git/GitHub 适配器"表述被移除。

### F-C3 ｜ 策略完整性 + 可信根 ｜ 目标 §15 / §24.2
- **最终措辞方向**：政策包使用签名或由 OS 保护的可信哈希锚点；信任根不得与可修改政策文件同权限域；政策签名或可信锚点校验失败后进入 `POLICY_INTEGRITY_SAFE_MODE`；safe mode 仅允许本地 A0（查看本地状态、查看审计记录、导出脱敏诊断、由人工修复或重新签署政策）；默认禁止所有外部网络请求、模型调用、CLI 命令执行、Git 写入、浏览器自动化和桌面自动化；不自动切换备用适配器、不自动恢复 A1–A5；只有可信政策重新加载并通过校验后才能退出 safe mode。
- **验收标准**：篡改政策包后 ARH 启动即拒 / 进 `POLICY_INTEGRITY_SAFE_MODE` 且拒绝 A2+；自动化测试含"政策被改 → 启动失败"用例；篡改政策包后，任何外部 API、模型、CLI、浏览器或 UIA 调用均被拒绝，仅本地 A0 诊断可用。

### F-C4 ｜ 持久化计数器 ｜ 目标 §9 / §12
- **最终措辞方向**：所有 message_id、SHA、尝试次数、修复轮数、仲裁次数、备用切换次数、预算消耗均持久化于事件账本/SQLite，重启后不归零。
- **验收标准**：注入"处理中途杀进程"故障，重启后同 SHA/消息不重复、循环计数延续而非归零。

### F-C5 ｜ 授权天花板（修正） ｜ 目标 §10.2 / §15.1
- **最终措辞方向**：不把插件自报 `authorization_tier_ceiling` 当安全边界。分三层：插件上报技术能力 / 政策引擎配置允许上限 / 能力令牌本次最小权限。最终授权上限由 ARH 政策引擎保存与裁决，插件、协调器、模型均无权自提。
- **验收标准**：向政策授权上限为 A2 的适配器发 A4 动作 → 政策引擎拒绝留痕；合约测试覆盖越界拒绝。

### F-C6 ｜ 政策缺口终态 + 默认决策阶梯 ｜ 目标 §4.3 / §8.2 / 附录 A
- **最终措辞方向**：新增 `BLOCKED_NEEDS_POLICY_DECISION` 前先执行阶梯：① 可安全跳过则跳过 ② 有保守默认值则采用 ③ 有等价备用则切换 ④ 可降级完成则降级 ⑤ 才进入该终态。系统须同时生成建议新增政策规则，不得仅把选择题延迟到次日。
- **验收标准**：构造政策未覆盖非安全情景，系统走阶梯后进入终态且仅在次日摘要出现，并附建议政策规则；运行期无弹窗。

### F-C7 ｜ 非代码证据推迟 ｜ 目标 §18.2
- **最终措辞方向**：接受方案 A。MVP 仅代码开发闭环；文档/研究/营销/数据分析工作流改为未来候选，待各自证据类型（结构化 schema 校验、外部 API 状态查询、人工签名确认等）定义后接入。
- **验收标准**：V1.1 不再暗示 MVP 同时覆盖代码与非代码闭环。

### F-C8 ｜ Phase 1 备份/恢复 ｜ 目标 §12
- **最终措辞方向**：Phase 1 必须完成 SQLite 一致性备份、WAL checkpoint 策略、产物备份、恢复演练、事件重放一致性校验。外部不可变审计锚定归 Team/Enterprise，不阻塞 Phase 1。
- **验收标准**：文档含可执行备份/恢复步骤与重放校验；审计锚定高档位要求被显式记录。

### F-C20 ｜ 子代理权限继承（由 P2 提升 P0） ｜ 目标 §4.2
- **最终措辞方向**：worker → 子代理的权限、预算、路径、域名、凭据范围均不得高于父代理；子代理继承父档位上限，禁止隐性提权。
- **验收标准**：构造子代理请求越权场景 → 被政策引擎拒绝并留痕。

### F-R2-01 ｜ 技术栈升级 ｜ 目标 §13.3
- **最终措辞方向**：`.NET 8` 改为 `.NET 10 LTS`（支持至 2028-11；.NET 8 于 2026-11 EOL）。核心/API、托盘与 Runner 统一 .NET 10 LTS。
- **验收标准**：全文技术建议统一为 .NET 10 LTS，无残留 .NET 8 引用。

### F-R2-02 ｜ 适配器优先级 ｜ 目标 §10.1 / §14 / 新增原则
- **最终措辞方向**：统一优先级 `Official API/SDK > CLI > Webhook > Playwright DOM > UIA > Visual/Coordinate Automation`；不得因桌面 UI"看得见"优先 UIA；GUI 通道不得作为无人值守工作流唯一完成路径。
- **验收标准**：V1.1 含该优先级表并贯穿适配器选型；任何"优先 UIA 因可见"表述被移除。

### F-R2-04 ｜ 取消 exactly-once 承诺 ｜ 目标 §10.2 / §16.3
- **最终措辞方向**：删除/废弃 `supports_exactly_once_claim`；改为 `supports_idempotency_key`、`supports_status_query`、`supports_external_correlation_id`、`supports_reconciliation`、`delivery_semantics`。ARH 对任意外部系统只保证"至少一次尝试、幂等优先、状态对账、无法确认时安全停止"，不得宣传跨任意第三方的 exactly-once。
- **验收标准**：协议能力清单无 `supports_exactly_once_claim`；文档定义交付语义字段与"安全停止"兜底。

### F-R2-05 ｜ 事务型 Outbox ｜ 目标 §11 / §12.2
- **最终措辞方向**：Outbox 保证内部状态变更与"待投递意图"在同一数据库事务中提交；外部动作记录至少区分 `PREPARED` / `DISPATCHING` / `ACKNOWLEDGED` / `VERIFIED` / `RECONCILIATION_REQUIRED`；崩溃恢复后不得盲目重发；若适配器支持 idempotency key、external correlation ID 或 status query，必须先查询并对账；若无法确定动作是否已经执行，进入 `RECONCILIATION_REQUIRED` 或对应安全终态；不得宣称 Outbox 能对任意第三方外部副作用实现 exactly-once。
- **验收标准**：注入外部动作执行前、执行中、执行后未写回等崩溃点；恢复后系统不盲目重发，能够对账的完成对账，无法确认的进入 `RECONCILIATION_REQUIRED`，不得伪造成功或保证绝对不重复。

### F-R2-06 ｜ 浏览器凭据 ｜ 目标 §14.2 / §15.2
- **最终措辞方向**：Playwright storage state、Cookie、localStorage、浏览器配置目录、会话令牌全部视同凭据：不入 Git、不入普通产物仓库、不出现于 prompt/日志/截图；使用独立受保护目录；支持撤销/过期/清理。
- **验收标准**：仓库与产物扫描无浏览器凭据泄露；凭据隔离目录策略文档化。

### F-R2-09 ｜ 状态权威与政策裁决权威 ｜ 目标 §6 / §24.3 / 新增
- **最终措辞方向**：事件账本是工作流事实、外部动作意图、投递结果和审计事件的唯一 Source of Truth；当前状态视图、仪表盘和查询表均为可从事件账本重建的派生 projection，不得反向覆盖事件事实；已签名政策包是授权边界与默认决策规则的 Source of Truth；政策引擎是权限、预算、工具路由和发布门禁的唯一裁决者，但不保存或伪造工作流事实；外部 worker / reviewer / messenger / repository / 模型只能提交声明、结果与证据，不能直接改写权威状态或政策。
- **验收标准**：删除 projection 后可从事件账本重建相同状态；外部工具返回"完成"只能生成待验证事件，不能直接把任务改为 `COMPLETED_APPROVED`；政策引擎的决策必须引用政策版本、输入事件和决策理由。

## P1 — 本版补强

### F-C9 ｜ 默认数值与多层预算 ｜ 目标 §9
- **最终措辞方向**：最大执行时间按工作流模板设置，须有默认值与绝对上限；预算分工作流/每日/全局三层。
- **验收标准**：§9 上限项均有默认数值；三层预算齐备。

### F-C10 ｜ 无备用默认终态 + 切换抖动防护 ｜ 目标 §4.3 / §16.1
- **最终措辞方向**：角色无任何备用适配器时主失效直接进对应安全终态；增最大切换尝试次数（默认 3）+ 切换退避。
- **验收标准**：构造"无备用"与"双坏备用"故障，分别进安全终态 / 有限次切换后终态。

### F-C11 ｜ 严重度无法裁决规则（修正） ｜ 目标 §7.6
- **最终措辞方向**：若同一 finding 的可信严重度候选中包含 `CRITICAL` 或 `HIGH`，且证据验证无法排除该高风险候选，则发布门禁阻断；只有全部可信候选严重度都低于发布门禁，才允许进入 `DEFERRED`；证据不足不能自动降级严重度；`DEFERRED` 必须记录候选严重度、未决原因和重新评估条件。
- **验收标准**（三类测试）：① `HIGH` 与 `LOW` 冲突且无法排除 HIGH → 阻断；② `MEDIUM` 与 `LOW` 冲突且均低于门禁 → 可 `DEFERRED`；③ 出现新确定性证据 → 重新计算严重度并更新 finding 状态。

### F-C12 ｜ 发布只认外部证据 ｜ 目标 §7.6 / §8
- **最终措辞方向**：release_gate 仅接受 evidence_verifier 确定性输出或外部可验证状态；禁止以任何角色纯文本"完成"作为晋级依据。
- **验收标准**：worker 返回"完成"但 verifier 未过 → 必阻断。

### F-C13 ｜ 插件崩溃退避 + 能力降级 ｜ 目标 §10.4 / §16
- **最终措辞方向**：同插件 N 分钟内崩溃超 M 次则停用告警不无限重启；运行中适配器丢失关键能力 → 对在途 job 触发重路由或安全终态。
- **验收标准**：崩溃风暴被退避遏制；运行时降级 → 在途 job 正确反应。

### F-C14 ｜ verifier 沙箱出网 + 脱敏方法 ｜ 目标 §15 / §12.3
- **最终措辞方向**：evidence_verifier 沙箱默认阻断出网、限制文件系统白名单；截图脱敏方法文档化（敏感字段替换 + 默认开启 + 关闭需显式策略授权并留审计）。
- **验收标准**：沙箱出网测试被阻断；脱敏方法可配置。

### F-C15 ｜ SDK 契约文档化（Phase 0） ｜ 目标 §10 / §20
- **最终措辞方向**：SDK 生命周期（load → negotiate → serve → shutdown）、错误码、取消、恢复、健康检查必须在 Phase 0 文档化（ARCHITECTURE/PROTOCOL 指针或骨架）。
- **验收标准**：存在 SDK 契约文档或明确指针；Phase 0 有可勾选验收清单。

### F-C16 ｜ SBOM 与 n8n 许可精核 ｜ 目标 §23.3
- **最终措辞方向**：指定 SBOM 标准（CycloneDX）与生成时机（CI 门禁）；核对 n8n 实际许可（Sustainable Use License）具体条款，替换笼统"fair-code"。
- **验收标准**：V1.1 含 SBOM 标准与 n8n 许可精确表述。

### F-C17 ｜ 并行/依赖模型 + 可观测性 ｜ 目标 §13 / §17
- **最终措辞方向**：声明"当前仅单 job，并行为后续"；定义结构化指标/SLO/告警阈值（不止 UI 字段）。
- **验收标准**：架构声明并行模型；可观测性需求文档化。

### F-R2-03 ｜ Qoder 实现路线 ｜ 目标 §18.1 / §19 / Phase 2
- **最终措辞方向**：Phase 2 以本地 Qoder CLI 和稳定 Agent SDK 为基线，优先验证 Qoder CLI / headless / Agent SDK / 结构化输出 / 流式事件 / 取消恢复 / 工具权限 / 进程退出码 / 会话关联；任何标记为 experimental / unstable 的云端能力不得成为正式闭环的唯一依赖；桌面 UIA 仅作 CLI/SDK 无法覆盖功能的补充通道。
- **验收标准**：Phase 2 探针清单含上述项；Qoder 首个适配器选型以 CLI/SDK 优先。

### F-R2-07 ｜ WebView2 边界 ｜ 目标 §13.3 / §14.2
- **最终措辞方向**：WebView2 仅用于 ARH 管理界面/仪表盘；第三方网页自动化由独立 Playwright Browser Runner 执行，专用浏览器上下文，不与 ARH UI WebView2 或用户个人浏览器混用。
- **验收标准**：V1.1 明确两者隔离；无混用表述。

### F-R2-08 ｜ 桌面自动化门禁 ｜ 目标 §14.1 / Phase 2
- **最终措辞方向**：UIA 探针必须验证锁屏、UAC、权限级别不同、窗口失焦、RDP 最小化/断开、应用更新、多窗口同名、输入法与剪贴板污染、目标进程崩溃、UI 元素变化；任一可能导致错误窗口写入时立即停止桌面通道，不允许盲输。
- **验收标准**：Phase 2 门禁含上述全项；盲输被禁止。
